    Farmburgh

    Indie Game


    Farmburgh is a point-and-click farming simulator that I independently developed and released on Steam. Prior to Farmburgh, I had played around in Unity and Unreal, creating simple prototypes and learning what I could. I felt that it was time to actually develop a game from start to finish in order to get a small taste of the industry. Farmburgh was developed in Unity using C# and all of the programming and game design was done by me. I quite enjoyed this process, especially coming up with creative features and overcoming difficult problems.



    Why did I create Farmburgh?

    Recently, I have been wanting to start a career in software development or game development. While my minor in college as computer science and my major of statistics had of programming in it, there really wasn’t anything about me that stands out in the sea of engineers. So, I developed Farmburgh in order to get an edge in the field (if you’re reading this, it obviously worked!) and to hone up some of my skills so that I can manage in an actual job. Not only that, but I am really passionate about video games. It has always been my dream to make a video game and I cannot believe I’ve finally been able to release one.



    How it works.

    Farmburgh’s core gameplay revolves a tile based grid. Each tile is it’s own GameObject and stores it’s own location and whether or not it contains an object. When and object such as and animal or building is placed, the tile that is selected will calculate what other tiles will be needed for that object using an algorithm. a TileManager is also being used to save and load tile data as well as making sure the tiles follow the rules. There are various objects that are able to be placed on the farm such as crops, animals, buildings and decorations. Each of these objects has it’s own class derived from a “TileableObject” class. They also have different things they are able to do once placed. Crops have a set time in which they grow and can be harvest while buildings are able to store crafting stations to craft various goods. Saving and loading player and game data was a big focus for me on this game. On the prototypes of different games I have tinkered with in the past, I have always hard-coded game data right into the scripts and I’ve never attempted to save player data. From the get-go, I wanted to attempt to do something right. After a few attempts at raw CSV files and JSON, I decide to go with SQLite. I knew a bit of SQL from my time as a stat major in college and I wanted to hone up those skills to use as a career option, so this was the perfect choice. I created the game data in an SQL editor outside of the game, such as crop names, growth times, etc. I then created a DataManager class as a singleton class and created a bunch of dictionaries to store the loaded data once the game started, for quick and easy access. The player’s save data also saves onto a local SQLite database using it’s own manager classes. You can check out some of these classes on the Github repo.
